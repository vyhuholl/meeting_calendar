## Requirements
### Requirement: Employee directory listing
Система SHALL предоставлять список всех сотрудников через API. Каждый сотрудник
имеет `id` и `name`. Список используется как для выбора «кто я», так и для выбора
участников встречи.

#### Scenario: Список сотрудников возвращается
- **WHEN** клиент запрашивает `GET /employees`
- **THEN** система возвращает HTTP 200 и массив объектов вида `{id, name}`

#### Scenario: Пустой справочник
- **WHEN** в базе нет ни одного сотрудника и клиент запрашивает `GET /employees`
- **THEN** система возвращает HTTP 200 и пустой массив

### Requirement: Employee identity selection ("кто я")
Система SHALL использовать выбор сотрудника из списка вместо настоящей
авторизации. UI SHALL показывать выпадающий список сотрудников; идентификация
происходит выбором, без логина и пароля.

#### Scenario: UI предлагает выбрать сотрудника
- **WHEN** пользователь открывает приложение
- **THEN** UI показывает выпадающий список сотрудников, полученный из `GET /employees`,
  для выбора «кто я»

### Requirement: Employee directory seeding
Система SHALL при пустой базе обеспечивать наличие набора сотрудников, чтобы
приложением можно было пользоваться сразу (без ручного создания через SQL).

#### Scenario: Начальное наполнение при первом запуске
- **WHEN** приложение стартует с пустой таблицей `employees`
- **THEN** система создаёт предопределённый набор сотрудников идемпотентно
  (повторный старт не создаёт дубликатов)
