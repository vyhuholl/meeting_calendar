## Context

Тестовое задание: планировщик встреч. Репозиторий greenfield — есть только
каркас (`pyproject.toml`, `Makefile`, `Dockerfile`, пустой `app/`). Стек задан
в `CLAUDE.md`: FastAPI (вся бизнес-логика), Streamlit (тонкий клиент к API),
SQLite + SQLAlchemy, uv, Docker. Ключевое ограничение — соразмерность: это
тестовое, не надо оверинжинирить (без SPA, Postgres, Redis, воркеров, Alembic).

Ядро задачи — запрет пересекающихся бронирований: нельзя создать встречу, если у
кого-то из участников уже есть встреча в пересекающееся время. Правило ломать
нельзя.

## Goals / Non-Goals

**Goals:**
- Корректная и протестированная логика пересечения полуоткрытых интервалов
  `[starts_at, ends_at)`.
- Атомарная проверка занятости участников + вставка в одной транзакции.
- Понятный отказ 409 с указанием, кто и когда занят.
- Просмотр расписания на день/неделю, фильтр по сотруднику.
- Тонкий Streamlit-клиент; API и UI — отдельные процессы, base URL через env.
- `make validate` (ruff + mypy) и `make test` (pytest, покрытие) проходят.

**Non-Goals:**
- Настоящая авторизация (только выбор «кто я» из списка).
- Редактирование/удаление встреч, приглашения, статусы участия, повторяющиеся
  встречи, уведомления — вне объёма, если не попросят явно.
- Миграции (Alembic), другая СУБД, кэш, очереди, фоновые задачи.
- Мультитенантность, роли, права доступа.

## Decisions

### D1. Слои приложения (структура)
Раскладка по `CLAUDE.md`:
```
app/
  main.py        # создание FastAPI, create_all на старте, seed сотрудников, подключение роутеров
  config.py      # настройки: DATABASE_URL (env), TZ-соглашения
  db.py          # engine, SessionLocal, Base, get_db (зависимость)
  models.py      # Employee, Meeting, MeetingParticipant (M2M)
  schemas.py     # Pydantic: EmployeeOut, MeetingCreate, MeetingOut, ConflictError
  conflicts.py   # чистая функция intervals_overlap + поиск конфликтов участников
  crud.py        # операции с БД, включая транзакционное создание встречи
  routers/
    employees.py # GET /employees
    meetings.py  # POST /meetings, GET /meetings (расписание день/неделя, фильтр)
ui/
  app.py         # Streamlit: выбор «кто я», форма встречи, просмотр расписания
tests/
  test_conflicts.py  # граничные случаи пересечений (обязательно)
  test_meetings.py   # API: создание, 409, валидация, расписание
```
*Альтернатива*: всё в одном `main.py`. Отклонено — разделение логики/роутов/CRUD
даёт тестируемость без заметного оверхеда для этого объёма.

### D2. Функция пересечения — чистая и отдельно тестируемая
`intervals_overlap(s1, e1, s2, e2) -> bool` возвращает `s1 < e2 and s2 < e1`.
Полуоткрытые интервалы: впритык не конфликтует. Функция не знает про БД —
тестируется таблицей граничных случаев (впритык, вложенный, частичное слева/справа,
одинаковые границы, непересекающиеся).
*Альтернатива*: считать конфликт прямо в SQL-запросе. Используем SQL для отбора
кандидатов (`WHERE starts_at < :new_end AND :new_start < ends_at`), но саму
семантику дублируем чистой функцией для юнит-тестов и ясности.

### D3. Транзакционная проверка + вставка (защита от гонки)
Создание встречи в одной транзакции:
1. `INSERT` встречи, привязка участников (или сбор их id).
2. `SELECT` пересекающихся встреч по участникам (`s1 < e2 AND s2 < e1`).
3. Если найден конфликт — `rollback` и HTTP 409; иначе `commit`.

В SQLite запись сериализуется (блокировка на запись), поэтому отдельного механизма
блокировок не нужно — этого достаточно по условию. Порядок «проверить, затем
вставить» внутри одной транзакции исключает окно гонки.
*Альтернатива*: проверка вне транзакции с последующей вставкой — отклонено, есть
окно гонки (двойное бронирование).

### D4. Модель данных и M2M
- `employees(id, name)`
- `meetings(id, title, starts_at, ends_at)` — время в UTC (naive/aware — см. D6)
- `meeting_participants(meeting_id, employee_id)` — ассоциативная таблица, PK по
  паре, FK на обе стороны. Минимум 2 участника — валидация на уровне Pydantic-схемы.

### D5. Отбор конфликтов одним запросом
Для набора `participant_ids` и интервала `[new_start, new_end)`:
`SELECT` встреч, где есть общий участник И `starts_at < new_end AND new_start < ends_at`.
Из результата формируем сообщение 409: имя(имена) занятого сотрудника и
конфликтующий интервал.

### D6. Время в UTC
Храним `starts_at`/`ends_at` в UTC. API принимает и отдаёт ISO-8601 (aware,
UTC). Сравнение — в UTC. Streamlit показывает в локальной TZ (конвертация на
клиенте). Требуем `starts_at < ends_at` (валидация схемы).

### D7. Схема БД на старте, seed сотрудников
`Base.metadata.create_all(engine)` на старте (идемпотентно, без Alembic). При
пустой таблице `employees` — идемпотентный seed предопределённого набора имён,
чтобы приложением можно было пользоваться сразу.

### D8. Streamlit — тонкий клиент
UI не ходит в БД, только в API по HTTP через `API_BASE_URL` (env, дефолт
`http://localhost:8000`). Экран: выбор «кто я» (dropdown из `GET /employees`),
форма создания встречи (title, дата/время начала и конца, мультиселект
участников), просмотр расписания на день/неделю с фильтром по сотруднику. Ошибку
409 показываем человекочитаемо (кто и когда занят). HTTP-клиент — `httpx` или
`requests` (добавить через `uv add`).

### D9. Запуск API и UI как отдельных процессов
Dockerfile уже запускает API (uvicorn). UI запускается отдельным процессом/
контейнером командой `streamlit run ui/app.py`. Для локали — две команды через
`uv run`. `docker-compose` можно добавить при желании, но не обязателен для
тестового.

## Risks / Trade-offs

- **[Гонка при параллельных бронированиях]** → проверка и вставка в одной
  транзакции; SQLite сериализует запись. Достаточно по условию.
- **[Путаница TZ: naive vs aware]** → фиксируем: API работает в UTC (aware),
  конвертация в локаль только в UI. Тесты используют UTC.
- **[SQLite и параллелизм записи]** → приемлемо для тестового; при росте нагрузки
  мигрировать на Postgres (вне объёма).
- **[Оверинжиниринг]** → сознательно держим минимальный набор слоёв; никаких SPA/
  Redis/воркеров.
- **[Расписание: какие встречи попадают в день/неделю]** → включаем встречи,
  чей интервал пересекается с окном (а не только начинается внутри), чтобы длинные
  встречи не терялись.
- **[Целостность M2M]** → FK + валидация «минимум 2 участника» и существования
  `employee_id` на входе.

## Migration Plan

Greenfield, миграций нет. Развёртывание: `uv sync`, старт API (`uvicorn
app.main:app`), старт UI (`streamlit run ui/app.py`) с `API_BASE_URL`. Схема
создаётся на старте, сотрудники засеиваются идемпотентно. Откат — удалить файл
SQLite (данные тестовые). Гейты: `make validate`, `make test`.
