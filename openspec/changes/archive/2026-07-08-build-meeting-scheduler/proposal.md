## Why

Планировщик встреч решает конкретную проблему координации: сотруднику нужно
назначить встречу нескольким участникам и быть уверенным, что ни у кого из них
это время уже не занято. Ручная сверка расписаний неизбежно приводит к
двойным бронированиям. Проект greenfield — сейчас в репозитории только каркас
(пустой `app/`, есть `pyproject.toml`, `Makefile`, `Dockerfile`), поэтому весь
продукт реализуется одним change.

## What Changes

- Новый бэкенд на **FastAPI**: модели, схемы, CRUD, логика конфликтов, роуты.
- Хранилище **SQLite + SQLAlchemy**; схема создаётся идемпотентно на старте через
  `Base.metadata.create_all(engine)` (без Alembic).
- Справочник сотрудников и выбор «кто я» (без настоящей авторизации — выпадающий
  список).
- Создание встречи с обязательной проверкой пересечений: если хотя бы один
  участник занят в пересекающемся интервале — отказ **HTTP 409** с указанием, кто
  именно и когда занят. Проверка занятости и вставка — в одной транзакции.
- Просмотр расписания на **день/неделю**, с возможностью отфильтровать по
  сотруднику.
- Тонкий фронтенд на **Streamlit**: форма создания встречи и просмотр расписания;
  ходит в API по HTTP (`API_BASE_URL`, дефолт `http://localhost:8000`).
- Тесты (**pytest**) — в первую очередь на функцию пересечения интервалов
  (граничные случаи) и на отказ при конфликте.
- Контейнеризация: API и UI — отдельные процессы (Dockerfile уже есть, при
  необходимости дополняется командой запуска UI).

## Capabilities

### New Capabilities
- `employee-directory`: справочник сотрудников — список для выбора «кто я» и для
  назначения участников встречи.
- `meeting-scheduling`: создание встречи с участниками и **критическое** правило
  запрета пересечений (полуоткрытые интервалы `[starts_at, ends_at)`, отказ 409 с
  указанием виновника и времени, атомарная проверка+вставка).
- `schedule-viewing`: просмотр расписания на день/неделю с фильтром по сотруднику;
  время хранится в UTC, показывается в локальной TZ.

## Impact

- **Новый код**: `app/` (main, config/db, models, schemas, crud, conflicts, routers),
  `ui/` (Streamlit-клиент), `tests/` (логика пересечений и API).
- **Зависимости**: уже объявлены в `pyproject.toml` (fastapi, sqlalchemy,
  streamlit; dev: mypy, pytest, pytest-cov, ruff). При необходимости добавляется
  `uvicorn`/`httpx` через `uv add`.
- **Хранилище**: файл SQLite (путь через env, дефолт локальный файл); создаётся на
  старте приложения.
- **Инфраструктура**: `Dockerfile` (API), запуск UI как отдельного процесса;
  `make validate` и `make test` — обязательные гейты качества.
- **Домен**: ядро — функция пересечения интервалов и транзакционная проверка
  занятости; ломать это правило нельзя.
